config.toml sketch

[paths]

output_dir  = ".../Voice Notes"

[transcribe]

language = "en"

[synthesis]

note_lifecycle = "evergreen"

[git]

Output tree: _archive/{audio,transcripts}, notes/, _system/. All writes under output_dir; link_context_dir read-only; overlap guard forbids collisions.

How we'll verify it end-to-end

  • python -m voicevault run <file.m4a> → audio archived, verbatim transcript, concept note(s) in defined domains, resolving [[wikilinks]], one git commit, source untouched.
  • Add a made-up acronym to dictionary.md, record it, confirm it lands correctly in the final note (bias + normalize — see the mechanism note below).
  • Point audio_src at a plain folder with no Obsidian vault → still runs; links resolve among its own notes (folder-agnostic).
  • Open output_dir in Obsidian → links + backlinks resolve; re-run on a saved transcript to iterate without re-recording.
Mechanism & honesty on acronyms: whisper doesn't hard-match your written words. dictionary.md feeds hotwords/initial_prompt — the terms go into the decoder's context and raise the prior probability of that spelling/casing. It's soft biasing, not a guarantee: great for casing & homophones, shakier for a brand-new acronym spoken as letters. That's why there are two more layers: a deterministic alias → canonical normalize pass, and the synthesis LLM (which also sees the dictionary) correcting in context. The three layers together are what make the term reliably correct in the final note.

The refinement cycle — how you steer output

The whole point: guide the pipeline toward the notes you want without touching code. Three things make that loop tight — a cheap iteration path, a richer control plane than "just three files," and the right model for the reasoning.

1 · The fast loop — re-synthesize from saved transcripts

Because the verbatim transcript is archived, synthesis is replayable. You don't re-record or re-transcribe to iterate:

edit a control file → python -m voicevault resynth <transcript> → diff the note → repeat

Seconds per iteration, near-zero cost (small transcripts), no audio reprocessing. This is where you'll spend most steering effort.

2 · The control plane — is markdown enough?

Mostly yes — with two additions. Markdown is excellent for declarative vocab, structure, and style rules. What it can't express well is "sound like this" and "stop doing that." So the control plane is five markdown surfaces, not three:

dictionary.md

Vocab: spelling, acronyms, alias → canonical.

taxonomy.md

Structure: the domains notes get filed into.

synthesis-guide.md

Rules: style, linking, split-by-concept, evergreen.

examples/ new

Hand-curated gold-standard notes fed as few-shot anchors — the single strongest lever for matching your voice.

feedback.md new

A running "lessons" log ("too verbose", "prefer H2 over bullets") the synthesis prompt reads each run.

…direct note edits

Evergreen merge preserves your hand-edits on the next update (a real design rule, see risks).

Recommendation: start on Claude to set the quality bar and iterate the control files against a strong model — the backend stays swappable (OllamaBackend is a stub from day one). A mature control plane (guide + examples + feedback) is exactly what lets a local 32B+ model catch up later, so the markdown investment is also your migration path off the cloud.
